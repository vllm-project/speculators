# Train a Speculator Model

Interactive skill for training speculative decoding draft models. When invoked, guide the user through the full pipeline step by step.

______________________________________________________________________

## Instructions for Claude

When this skill is invoked, follow this interactive flow. Use `AskUserQuestion` to collect inputs before running anything. Confirm before executing each step.

### Phase 1: Gather Configuration

Ask the user these questions (use AskUserQuestion, batch where sensible):

**1. Target model**

- "Which model do you want to accelerate?"
- Common choices: `Qwen/Qwen3-8B`, `meta-llama/Llama-3.1-8B-Instruct`, `Qwen/Qwen3.5-9B`
- Accept any HuggingFace model ID or local path

**2. Algorithm**

- "Which speculative decoding algorithm?"
- Options:
  - **Eagle-3** (Recommended) — Autoregressive drafting, most mature tooling and vLLM support
  - **P-EAGLE** — Parallel multi-token prediction, extends Eagle-3 with COD sampling
  - **DFlash** — Block-based single-pass prediction
  - **MTP** — Finetune native MTP head (only for models with MTP layers like Qwen3-Next, Qwen3.5)

**3. Training dataset**

- "What training data?"
- Options:
  - `sharegpt` — Built-in ShareGPT dataset
  - `ultrachat` — Built-in UltraChat dataset
  - HuggingFace dataset — User provides a HuggingFace dataset ID (e.g., `org/dataset-name`). Pass the ID directly to `--data`.
  - Local file — User provides path to a local `.jsonl` file
- Note: MTP requires data generated by the target model. If MTP is selected and user doesn't have regenerated data, offer to run response regeneration first.

**4. Training mode**

- "Online or offline training?"
- Options:
  - **Online** (Recommended) — Hidden states generated on-the-fly. Simpler, lower disk usage, but needs GPUs split between vLLM and training simultaneously.
  - **Offline** — Pre-generate hidden states, then train. GPUs reused sequentially. Better when GPU resources are limited.
  - **Hybrid** — Generate and cache on first epoch, reuse on subsequent epochs.

**5. GPU setup**

- "Which GPUs are available? How should they be split?"
- For online: need separate GPUs for vLLM and training (e.g., vLLM on 0,1 and training on 2,3)
- For offline: same GPUs reused sequentially (e.g., 0,1 for both)
- Ask for: vLLM GPU IDs, training GPU IDs, vLLM port (default 8000)

**6. Training hyperparameters** (offer sensible defaults, let user override)

- Max samples: 5000 (good for sanity check; 50K+ for production)
- Sequence length: 8192
- Epochs: 5
- Learning rate: algorithm-dependent defaults (Eagle-3: 1e-4, P-EAGLE: 6e-4, DFlash: 3e-4, MTP: 1e-4)
- Output directory: `./output`

**7. Advanced settings** (optional)

- After collecting the core configuration, ask the user if they want to customize advanced settings.
- If yes, present an AskUserQuestion with the following multi-select options. Show algorithm-appropriate defaults inline. Let the user pick which settings to customize, then follow up for values.
- Advanced settings to offer:
  - **Loss function** — `kl_div` (default) or `ce` (cross-entropy)
  - **Noise augmentation** — std dev for noise injection (default: 0.05). Set to 0 to disable.
  - **LR scheduler** — `linear` (default for Eagle-3/DFlash/MTP), `cosine` (default for P-EAGLE), or `none`
  - **Warmup steps** — number of LR warmup steps (default: none)
  - **Draft architecture** — `llama` (default) or `qwen3`. Should match target model family.
  - **Draft vocab size** — reduced vocabulary for the draft head (default: 32000 for Eagle-3/DFlash, full vocab for P-EAGLE/MTP)
  - **Number of draft layers** — override the algorithm default (Eagle-3: 1, P-EAGLE: 4, DFlash: 5)
  - **Checkpoint frequency** — save every N epochs (default: 1)
  - **Resume from checkpoint** — path to a pretrained draft model (`--from-pretrained`) for continued training
  - **Logging** — enable `wandb`, `tensorboard`, or `trackio` experiment tracking
  - **Run name** — experiment name for the logger
  - **Random seed** — default: 42
  - **vLLM port** — default: 8000
- If the user declines advanced settings, use all defaults.

### Phase 2: Generate and Run the Training Script

After collecting all inputs, assemble the full training pipeline. Present the complete script to the user for review before running.

The pipeline has these steps (run each with user confirmation):

**Step 1: Prepare data**

```bash
python scripts/prepare_data.py \
    --model <MODEL> \
    --data <DATASET> \
    --output <OUTPUT_DIR> \
    --max-samples <MAX_SAMPLES> \
    --seq-length <SEQ_LENGTH>
```

**Step 2: Launch vLLM server** (run in background)

```bash
CUDA_VISIBLE_DEVICES=<VLLM_GPUS> python scripts/launch_vllm.py <MODEL> \
    [--target-layer-ids <IDS>]  \  # DFlash/MTP only
    [--hidden-states-path <PATH>] \  # P-EAGLE only
    -- --data-parallel-size <DP_SIZE> --port <PORT>
```

Wait for health check: `curl -sf http://localhost:<PORT>/health`

**Step 3a (offline only): Generate hidden states**

```bash
python scripts/data_generation_offline.py \
    --preprocessed-data <OUTPUT_DIR> \
    --endpoint http://localhost:<PORT>/v1 \
    --output <OUTPUT_DIR>/hidden_states \
    --concurrency 32 \
    --validate-outputs
```

Then stop vLLM to free GPUs.

**Step 3b: Train**

```bash
CUDA_VISIBLE_DEVICES=<TRAIN_GPUS> torchrun \
    --standalone --nproc_per_node <NUM_TRAIN_GPUS> \
    scripts/train.py \
    --verifier-name-or-path <MODEL> \
    --data-path <OUTPUT_DIR> \
    --vllm-endpoint http://localhost:<PORT>/v1 \    # online/hybrid only
    --hidden-states-path <OUTPUT_DIR>/hidden_states \  # offline/hybrid only
    --save-path <OUTPUT_DIR>/checkpoints \
    --epochs <EPOCHS> \
    --lr <LR> \
    --total-seq-len <SEQ_LENGTH> \
    <ALGORITHM_FLAGS> \
    --on-missing <generate|raise> \
    --on-generate <delete|cache>
```

For single-GPU training, use `python scripts/train.py` instead of `torchrun`.

**Step 4 (MTP only): Stitch weights**

```bash
python scripts/stitch_mtp.py \
    <OUTPUT_DIR>/checkpoints/checkpoint_best \
    <MODEL> \
    --output-path <OUTPUT_DIR>/stitched
```

### Phase 3: Post-Training

After training completes, tell the user:

1. Where checkpoints are saved
2. How to deploy:
   ```bash
   vllm serve <CHECKPOINT_PATH>
   ```
3. How to evaluate:
   ```bash
   cd scripts/evaluate && pip install -r requirements.txt
   python evaluate.py throughput --target http://localhost:8000/v1
   ```
4. How to continue training from checkpoint:
   ```bash
   --from-pretrained <CHECKPOINT_PATH>
   ```

## Algorithm-Specific Flags

### Eagle-3

No extra flags required. Optional:

- `--draft-vocab-size 32000`
- `--num-layers 1`
- `--draft-arch llama`

### P-EAGLE

Required flags:

```
--speculator-type peagle
--num-layers 4
--num-depths 7
--down-sample-ratio 0.6
--down-sample-ratio-min 0.2
--no-norm-before-residual
--scheduler-type cosine
```

### DFlash

Required flags:

```
--speculator-type dflash
--num-layers 5
--block-size 8
--max-anchors 3072
--draft-vocab-size 32000
--target-layer-ids 2 18 33
```

`--target-layer-ids` must match what vLLM was launched with.

### MTP

Required flags:

```
--speculator-type mtp
--num-speculative-steps 3
--target-layer-ids 32
--step-weight-beta 0.6
```

MTP uses `python scripts/train.py` (single GPU, no torchrun). After training, run `scripts/stitch_mtp.py` to merge weights back. Deploy with: `vllm serve <stitched_path> --speculative-config '{"method":"mtp","num_speculative_tokens":3}' --no-enable-chunked-prefill`

## Key Training Arguments Reference

### Data & Hidden States

| Flag                   | Default                     | Description                                                                    |
| ---------------------- | --------------------------- | ------------------------------------------------------------------------------ |
| `--data-path`          | `./output`                  | Root directory with preprocessed dataset                                       |
| `--hidden-states-path` | `<data-path>/hidden_states` | Path to cached hidden states                                                   |
| `--vllm-endpoint`      | `http://localhost:8000/v1`  | vLLM server URL                                                                |
| `--on-missing`         | —                           | What to do when hidden states are missing: `generate`, `skip`, `warn`, `raise` |
| `--on-generate`        | —                           | After generating: `cache` (save to disk) or `delete` (discard)                 |

### Training

| Flag              | Default  | Description                     |
| ----------------- | -------- | ------------------------------- |
| `--epochs`        | `20`     | Training epochs                 |
| `--lr`            | `1e-4`   | Learning rate                   |
| `--total-seq-len` | `8192`   | Max sequence length per batch   |
| `--loss-fn`       | `kl_div` | Loss function: `kl_div` or `ce` |
| `--noise-std`     | `0.05`   | Noise augmentation std dev      |
| `--seed`          | `42`     | Random seed                     |

### Model Architecture

| Flag                 | Default  | Description                                           |
| -------------------- | -------- | ----------------------------------------------------- |
| `--speculator-type`  | `eagle3` | Algorithm: `eagle3`, `peagle`, `dflash`, `mtp`        |
| `--num-layers`       | `1`      | Number of draft transformer layers                    |
| `--draft-vocab-size` | —        | Reduced vocabulary size                               |

| `--from-pretrained`  | —        | Path to pretrained draft model for continued training |

| `--no-resume-from-checkpoint` | —                      | Start training from scratch               |

### Scheduler

| Flag                       | Default  | Description                              |

## GPU Planning

### Online Training

Requires GPUs split between vLLM and training simultaneously:

| Setup       | vLLM GPUs      | Training GPUs | Total |
| ----------- | -------------- | ------------- | ----- |
| Minimal     | 1              | 1             | 2     |
| Recommended | 2 (DP=2)       | 2             | 4     |
| Large model | 4 (TP=2, DP=2) | 2-4           | 6-8   |

### Offline Training

GPUs are reused sequentially (data generation first, then training):

| Setup       | GPUs | Notes                                      |
| ----------- | ---- | ------------------------------------------ |
| Minimal     | 2    | DP=2 for data gen, then reuse for training |
| Recommended | 2-4  | Higher concurrency during data gen         |

## Troubleshooting

### OOM during training

- Reduce `--total-seq-len` (try 4096)
- Reduce `--num-layers` (try 1)
- Use more GPUs with FSDP

### OOM during vLLM

- Pass `-- --gpu-memory-utilization 0.95` to `launch_vllm.py`
- Pass `-- --max-model-len 4096` to reduce KV cache
- Add more GPUs with tensor parallelism

### Training loss not decreasing

- Lower `--lr` (try 1e-5)
- Train longer with more `--epochs`
- Check data quality with `ls -lh ./output/`

### Bursty GPU utilization during online training

- Give more GPUs to vLLM (training is bottlenecked on hidden state generation)
- Switch to offline training to decouple data generation from training

### vLLM server not responding

- Verify health: `curl http://localhost:8000/health`
- Check vLLM logs for startup errors
- Ensure vLLM GPUs don't overlap with training GPUs
